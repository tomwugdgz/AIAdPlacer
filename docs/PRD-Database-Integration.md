# PRD: pDOOH 点位数据库集成

## 1. 产品概述

### 1.1 功能名称
pDOOH 点位数据库集成

### 1.2 产品目标
为 AIAdPlacer 提供统一的户外广告点位数据查询能力，支持按多维度筛选和检索，辅助广告投放决策。

### 1.3 产品价值
- **数据整合**：将分散的 7 类点位数据统一管理和查询
- **快速检索**：支持按城市/商圈/点位类型等多维度筛选
- **决策支持**：提供统计数据，辅助广告投放策略制定
- **系统集成**：通过 RESTful API 和 MCP 工具两种方式提供服务

### 1.4 目标用户
- 广告投放师：查询和筛选广告点位
- 媒体策划：分析点位分布和统计数据
- 销售团队：查询客户信息和联系方式
- AI Agent：通过 MCP 工具自动调用点位数据

## 3. 功能需求

### 3.1 P0 - 必须实现

#### FR-001: 查询所有表名和记录数
**描述**：系统应提供接口返回数据库中所有表的名称和各表的记录数。

**验收标准**：
- 给定数据库连接正常，当调用获取表列表接口，那么返回所有 7 张表的名称和记录数
- 响应时间 < 500ms

#### FR-002: 按条件查询点位数据
**描述**：系统应支持按省份、城市、区域、商圈等条件筛选点位数据。

**字段映射**：
| 字段 | 门禁点位 | 单元门点位 | 智能屏L9 | 智能屏202507 | 商场LED点位 | 道闸点位 |
|------|---------|-----------|---------|-------------|------------|---------|
| 省份 | 省份 | 省份 | - | - | 省份 | 省份 |
| 城市 | 城市 | 城市 | 城市 | 城市 | 城市 | 城市 |
| 区域 | 区域 | 区域 | - | - | 区域 | - |
| 商圈 | 商圈 | 商圈 | - | - | - | - |

**验收标准**：
- 给定有效的筛选条件，当调用查询接口，那么返回符合条件的点位列表
- 支持多条件组合查询（AND 逻辑）
- 对于不存在的字段，系统应忽略该条件而不报错

#### FR-003: 分页查询
**描述**：所有列表查询接口必须支持分页，避免一次性加载大量数据。

**参数**：
- `page`: 页码（从 1 开始），默认 1
- `page_size`: 每页记录数，默认 20，最大 100

**验收标准**：
- 给定页码和每页大小参数，当调用查询接口，那么返回指定页的数据和分页信息
- 返回格式包含：`data` (数据列表), `total` (总记录数), `page` (当前页), `page_size` (每页大小), `total_pages` (总页数)

---

### 3.2 P1 - 重要

#### FR-004: 价格范围筛选
**描述**：系统应支持按价格范围筛选点位（适用于有价格字段的表）。

**适用表**：
- 门禁点位（字段：`价格`）
- 单元门点位（字段：`价格`）
- 智能屏L9（字段：`制作费`）
- 智能屏202507（字段：`价格`）
- 商场LED点位（字段：`价格`）

**参数**：
- `price_min`: 最低价格（可选）
- `price_max`: 最高价格（可选）

**验收标准**：
- 给定价格范围，当调用查询接口，那么返回价格在区间内的点位
- 只传入 `price_min` 时，返回价格 >= 该值的点位
- 只传入 `price_max` 时，返回价格 <= 该值的点位

#### FR-005: 全文搜索
**描述**：系统应支持在指定表中全文搜索关键字。

**搜索字段**：
- 门禁点位：小区名称、备注
- 单元门点位：小区名称、备注
- 智能屏L9：宣传名称、上刊地址
- 智能屏202507：小区名称、详细地址
- 商场LED点位：商场名称、详细地址
- 道闸点位：小区名称、详细地址
- 客户通讯录：客户名称、联系人、备注

**参数**：
- `q`: 搜索关键字（必填）
- `fields`: 搜索字段列表（可选，默认搜索所有文本字段）

**验收标准**：
- 给定搜索关键字，当调用搜索接口，那么返回包含该关键字的记录
- 搜索应不区分大小写
- 支持模糊匹配（LIKE '%keyword%'）

#### FR-006: 获取统计数据
**描述**：系统应支持按城市/省份分组统计点位数据。

**统计维度**：
- 按省份分组：统计各省的点位数量
- 按城市分组：统计各城市的点位数量

**适用表**：门禁点位、单元门点位、智能屏L9、智能屏202507、商场LED点位、道闸点位

**验收标准**：
- 给定表名和分组字段，当调用统计接口，那么返回分组统计结果
- 返回格式：`[{"省份": "广东省", "count": 12345}, ...]`
- 结果按数量降序排列

### 3.3 P2 - 可选

#### FR-007: 按媒体类型获取点位
**描述**：系统应支持按媒体类型（门禁/单元门/智能屏/商场LED/道闸）获取点位。

**媒体类型映射**：
- 门禁：门禁点位
- 单元门：单元门点位
- 智能屏：智能屏L9 + 智能屏202507
- 商场LED：商场LED点位
- 道闸：道闸点位

**验收标准**：
- 给定媒体类型，当调用接口，那么返回对应表的点位数据
- 对于智能屏类型，应合并两个智能屏表的数据

#### FR-008: 搜索客户信息
**描述**：系统应支持在客户通讯录中搜索客户信息。

**搜索字段**：客户名称、联系人、联系电话、备注

**验收标准**：
- 给定搜索关键字，当调用客户搜索接口，那么返回匹配的客户列表
- 支持分页

## 4. 非功能需求

### 4.1 性能要求
- **查询响应时间**：分页查询 < 500ms（数据量 < 10 万条）
- **统计查询响应时间**：分组统计 < 1s
- **并发支持**：支持至少 10 个并发查询请求

### 4.2 安全性
- **数据库文件保护**：数据库文件 `qinlin_local.db` 不提交到 Git（已在 `.gitignore` 中排除）
- **SQL 注入防护**：所有查询参数必须使用参数化查询，禁止字符串拼接
- **访问控制**：API 接口应根据需要添加认证机制（由上游网关或中间件处理）

### 4.3 兼容性
- **数据库**：支持 SQLite3 数据库
- **Python 版本**：Python 3.8+
- **依赖库**：`sqlite3` (标准库，无需额外安装)

### 4.4 可维护性
- **日志记录**：记录慢查询（响应时间 > 1s）
- **错误处理**：统一的错误处理机制，返回清晰的错误信息
- **代码注释**：关键函数和复杂 SQL 语句应有注释

## 7. 技术方案（简要）

┌─────────────┐
│  Frontend    │
│  (React)     │

       │

### 7.2 数据库访问层
- **文件**：`backend/app/db_dao.py`
- **技术**：使用 `sqlite3` 标准库
- **关键点**：
  - 使用参数化查询防止 SQL 注入
  - 表名包含中文，查询时需要用双引号括起来
  - 动态构建 WHERE 子句，只添加非空的筛选条件

### 7.3 API 框架
- **文件**：`backend/app/db_api.py`
- **技术**：FastAPI Router
- **路由前缀**：`/api/v2/db`
- **关键点**：
  - 使用 Pydantic 模型验证请求参数
  - 统一的错误处理和响应格式
  - 支持 CORS（由主应用配置）

### 7.4 MCP 集成
- **文件**：`backend/app/mcp_server.py`
- **技术**：FastMCP
- **新增工具**：3 个（pdooh_query_pdooh_points, pdooh_get_point_stats, pdooh_search_clients）
- **关键点**：
  - 工具描述应清晰说明功能和参数
  - 调用现有的 DAO 函数，避免重复代码

## 8. 风险与依赖

### 8.1 风险

#### 风险 1: 数据库文件较大
- **描述**：数据库文件 27MB，需要确保不提交到 Git
- **影响**：占用仓库空间，可能泄露数据
- **缓解措施**：
  - 已在 `.gitignore` 中添加 `backend/data/*.db`
  - 在 README 中说明如何获取数据库文件
  - 提供数据库结构导出脚本（SQL dump）

#### 风险 2: 部分数据缺少经纬度坐标
- **描述**：门禁点位等表中部分记录的经纬度为 NULL
- **影响**：无法在地图上展示这些点位
- **缓解措施**：
  - 在文档中说明数据完整性情况
  - API 返回所有字段，由前端处理 NULL 值
  - 未来可通过地理编码补全坐标

#### 风险 3: 表名包含中文
- **描述**：SQLite 查询中文表名需要用双引号括起来
- **影响**：SQL 语句构建时需要特殊处理
- **缓解措施**：
  - 在 DAO 层统一处理表名引用
  - 添加单元测试验证中文表名查询

#### 风险 4: 数据库文件不存在
- **描述**：部署环境中可能缺少数据库文件
- **影响**：所有数据库相关功能无法使用
- **缓解措施**：
  - API 返回清晰的错误信息
  - 提供数据库文件下载或导入说明

### 8.2 依赖

#### 内部依赖
- **数据库文件**：`backend/data/qinlin_local.db` 必须存在
- **Python 环境**：Python 3.8+ 

#### 外部依赖
- **FastAPI**: Web 框架（已在项目中引入）
- **sqlite3**: Python 标准库，无需额外安装
- **FastMCP**: MCP Server 框架（已在项目中引入）

#### 前置任务
- 无（数据库文件已提供）
