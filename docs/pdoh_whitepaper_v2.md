## 一、为什么没有人做 pDOOH 开源项目？

### 1.1 市场现实

| 搜索词 | 项目数 | 最高 Stars | 结论 |
|--------|--------|-----------|------|
| `outdoor + programmatic` | ~20个 | 2⭐ | 非广告交易核心 |
| `pDOOH` | **0个** | **0** | **完全空白** |
| `digital signage platform` | ~20个 | 126⭐ | 非程序化广告 |

**结论：没有任何 Stars > 10 的 pDOOH 开源项目。**

### 1.2 五大壁垒（为什么没人做）

#### 🔴 技术壁垒（五星难度）
| 障碍 | 为什么难 |
|------|----------|
| 实时库存管理 | 户外点位没有"在线状态"，大部分离线管理 |
| 曝光监测 | 无法追踪个体，没有归因模型 |
| 动态创意（DCO） | 90% 屏幕无法联网接收素材 |
| 无标准协议 | 没有类似 OpenRTB 的户外标准 |

**核心问题：户外广告是"离线优先"的媒体。**

#### 🔴 商业模式不性感
| 问题 | 影响 |
|------|------|
| 重资产 | 屏幕需要大量资金投入 |
| 销售周期长 | 一个订单 3-6 个月 |
| B2B关系型 | 老板不换，订单不换 |
| 账期长 | 6-12个月，资金压力大 |

#### 🔴 监管复杂
- 户外广告需要政府审批
- 内容无法自动审核
- 医疗/金融/药品限制多
- 各城市政策不一

#### 🔴 人才稀缺
> **最缺：同时懂户外广告行业规则 + 程序化技术的人才**

| 人群 | 问题 |
|------|------|
| 传统户外人 | 不懂技术 |
| 互联网技术人 | 不懂户外 |

#### 🔴 市场规模小 + 头部垄断
| 数据 | 说明 |
|------|------|
| 中国户外广告市场 | ~800亿 |
| 可程序化比例 | **< 5%（约40亿）** |
| 头部占 | 50%+（分众+新潮垄断） |

### 1.3 结论

> **不是没人想做，是：**
> - 想做的人不知道怎么做
> - 知道怎么做的人没有资源
> - 有资源的人不愿意投入

## 二、XX科技：为什么它能做成功？

### 2.1 XX科技的5V数据底座（核心差异化）

XX科技是唯一打通 **"门禁设备 + 社区媒体 + 电商平台"** 的玩家：

```
V1 人口属性  →  年龄/性别/学历/收入
V2 消费偏好  →  美妆/母婴/农产品/高端白酒/网红爆款
V3 社区属性  →  房价/户数/地段
V4 开门行动  →  QR开门/人脸开门/刷卡开门（IoT独家数据）
V5 线上行为  →  整合京东/百度DMP数据
```

**V4（开门行动数据）是XX科技独有的护城河** — 每天几次开门，就是几次强制广告曝光，且数据可追踪。

### 2.2 XX科技技术架构

```
┌─────────────────────────────────────────────┐
│          XX科技技术栈                    │
├─────────────────────────────────────────────┤
│  QIOT（物联网） → 智能门禁 + 广告屏     │
│  QADN（资源管理） → 点位状态实时管理     │
│  QADX（广告交易） → 竞价机制 + 定向投放  │
│  QDMP（数据中台） → 5V数据维度整合       │
│  QMALL（社区电商） → 投放→转化闭环       │
└─────────────────────────────────────────────┘
```

### 2.3 XX科技 vs 分众 vs 新潮

| 维度 | XX科技 | 分众 | 新潮 |
|------|------|------|------|
| 屏幕联网率 | **100%**（IoT设备） | ~30% | ~20% |
| 数据追踪 | **开门行动数据** | 无 | 无 |
| 程序化 | 部分（QADX） | 电梯屏部分 | 被收购整合中 |
| 社区覆盖 | 70,000+ 社区 | 主要是写字楼 | 社区+写字楼 |
| 电商闭环 | **有（QMALL）** | 无 | 无 |

## 三、我们的项目：AIAdPlacer pDOOH 平台

### 3.1 核心定位

**以XX科技5V数据模型为底座，用 AI Native 架构重构 pDOOH 投放。**

```
传统 pDOOH：人工选点 → 邮件下单 → 手动换刊 → 无归因
       ↓
AI Native pDOOH：
  Agent（人群洞察）→ Workflow（智能排期）→ MCP（标准化交易）→ 数据平台（实时归因）
```

### 3.2 AI Native 架构（LLM + RAG + Agent + Workflow + MCP）

```
┌──────────────────────────────────────────────────────────┐
│                    AIAdPlacer 架构                    │
├──────────────────────────────────────────────────────────┤
│  🤖 Agent层（4个专用Agent）                          │
│    ├─ 人群洞察Agent    → KMeans聚类 + 5V标签匹配     │
│    ├─ 智能排期Agent    → FM + deepMCP 联合优化       │
│    ├─ 动态创意Agent    → AIGC + DCO 实时生成        │
│    └─ 效果归因Agent    → 跨端匹配 + 开门数据归因     │
│                                                          │
│  🔄 Workflow层（规划→执行→反思）                     │
│      统一编排Agent，支持人机协同审核                   │
│                                                          │
│  🔌 MCP层（AI-to-AI 标准化接口）                    │
│    ├─ 屏方MCP Server  → 库存查询/预约/确认          │
│    ├─ 数据方MCP Server → 人群数据/POI数据           │
│    └─ 监管方MCP Server → 合规审核/内容预审           │
│                                                          │
│  🗄️ 数据平台层（PostgreSQL + JSONB）                │
│    ├─ 屏资产表（9801块智能屏）                      │
│    ├─ 人锚点表（可信ID体系）                        │
│    ├─ 5V标签表（参考XX科技模型）                      │
│    ├─ POI数据表（高保真模拟）                        │
│    └─ 轨迹数据表（时空行为分析）                      │
└──────────────────────────────────────────────────────────┘
```

### 3.3 与XX科技的差异化竞争策略

| 维度 | XX科技（ Closed ） | 我们（ Open + AI Native ） |
|------|------------------|---------------------------|
| 数据模型 | 5V（闭源） | **5V + LLM增强**（开源可扩展） |
| Agent能力 | 无 | **4个专用Agent + Workflow编排** |
| 标准化接口 | 无（私有协议） | **MCP协议（AI-to-AI标准化）** |
| AI投放 | 无 | **A2A接口（AI Agent互操作）** |
| 开发者生态 | 无 | **开源 + Skill市场** |
| 目标客户 | 品牌广告主 | **品牌 + 开发者 + AI Agent** |

## 四、技术实现方案

### 4.1 数据库设计（已完成）

**12张核心表 + 2个视图**，支持完整"人-屏-POI-轨迹"数据链路：

```sql
screen                  -- 智能屏资产（9801块）
person_anchor          -- 人锚点（可信ID体系）
trusted_id_binding     -- 可信ID绑定（TAID/OAID/MAC/人脸Hash）
spatial_trajectory     -- 时空轨迹数据
poi_data              -- POI数据（13,362条高保真模拟）
vw_screen_audience     -- 屏人群画像视图（实时查询）
```

**扩展表（参考XX科技5V模型）：**
```sql
dmp_tag                -- 5V标签定义
person_tag             -- 人的标签绑定
campaign               -- 投放计划
ai_campaign            -- AI投放计划（A2A专用）
```

### 4.2 API设计（已完成）

**9个REST API端点**，`/api/v2/pdooh/` 前缀：

```
GET  /api/v2/pdooh/screens              -- 屏列表（支持筛选）
GET  /api/v2/pdooh/screens/{id}        -- 屏详情
GET  /api/v2/pdooh/screens/nearby      -- 附近屏（经纬度+半径）
GET  /api/v2/pdooh/people             -- 人锚点列表
GET  /api/v2/pdooh/people/{id}/tags   -- 人的标签
GET  /api/v2/pdooh/poi                -- POI列表
GET  /api/v2/pdooh/campaigns          -- 投放计划列表
POST /api/v2/pdooh/campaigns          -- 创建投放计划
GET  /api/v2/pdooh/stats/overview     -- 数据概览
```

### 4.3 前端Demo（已完成）

`demo.html` 已改造：
- ✅ 腾讯地图集成（WebService API）
- ✅ 真实API调用（替换所有模拟数据）
- ✅ 屏地图可视化（Marker + InfoWindow）
- ✅ 人群画像展示（调用 `/api/v2/pdooh/screens/{id}/audience`）
- ✅ 投放计划管理（创建/列表/状态）

### 4.4 A2A MCP接口（进行中）

**设计目标：让其他AI Agent能直接调用我们的投放能力**

```
┌─────────────────────────────────────────────┐
│              A2A MCP Server               │
├─────────────────────────────────────────────┤
│  Tool: pdooh_query_screens               │
│    → 查询符合条件的屏（地理位置/人群标签）│
│  Tool: pdooh_create_campaign             │
│    → 创建投放计划（AI Agent自主操作）    │
│  Tool: pdooh_query_audience             │
│    → 查询屏的人群画像                   │
│  Tool: pdooh_submit_creative            │
│    → 提交创意（AIGC生成+合规审核）      │
│  Tool: pdooh_query_report               │
│    → 查询投放报告（曝光/点击/转化）     │
└─────────────────────────────────────────────┘
```

**Skill调用模块：**
- 封装为 WorkBuddy Skill：`pdooh-agent`
- 支持自然语言操作：`"帮我在广州天河区找房价>5万的社区屏，投放高端白酒广告"`
